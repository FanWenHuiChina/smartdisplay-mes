**一句话记忆：**显示行业 MES 的核心不是“工单管理”，而是“Lot 在工艺路线和设备之间受控流转，并保留质量、设备、Recipe、物料、人员、时间的完整追溯链”。

## 2.2 关键业务对象

| **对象** | **含义** | **项目中怎么做** |
|----|----|----|
| Production Order 工单 | 生产任务，包含产品型号、计划数量、交期、优先级、目标产线。 | prod_order 表；状态：待下发、已释放、生产中、Hold、完工、关闭。 |
| Lot 批次 | MES 运行时核心对象，实际在工艺路线中流转。 | prod_lot 表；记录当前工序、当前设备、当前状态、是否返工。 |
| Panel / SN | 比 Lot 更细的屏体或模组序列号，用于单件追溯。 | prod_panel_unit 表；绑定 lot_id、当前状态、最终等级。 |
| Route 工艺路线 | 规定 Lot 必须经过哪些工序，以及顺序、可用设备、是否检测。 | md_route + md_route_step；支持版本号。 |
| Step 工序 | 清洗、涂胶、曝光、显影、蚀刻、剥膜、检测、封装、绑定、老化、终检等。 | md_process_step；配置是否关键工序、是否需要 Recipe、是否允许返工。 |
| Equipment 设备 | Lot 加工的执行载体，具备状态、能力、报警、保养信息。 | md_equipment；状态：Idle、Running、Alarm、Down、PM、Offline。 |
| Recipe 配方/参数包 | 设备加工参数，通常由产品型号 + 工序 + 设备决定。 | md_recipe + md_recipe_param；Track In 前必须校验。 |
| Defect 缺陷 | 质量异常分类，例如亮点、暗点、Mura、划伤、封装异常、绑定不良等。 | md_defect_code + quality_defect_record。 |
| Hold / Release | 异常控制动作：暂停 Lot，处理完成后放行。 | lot_hold_record；禁止 Hold 状态继续过站。 |
| Rework / Scrap | 返工或报废处置。 | lot_rework_record；返工需要选择返工路线。 |

## 2.3 五条最重要的业务规则

1.  工艺路线校验：Lot 只能进入当前 Route 配置的下一道合法工序，不能跳站、漏站、重复过站。

2.  设备能力校验：设备必须处于可用状态，并且支持当前工序；设备 Alarm/Down 时不允许 Track In。

3.  Recipe 校验：产品型号 + 工序 + 设备必须匹配有效 Recipe；Recipe 版本要进入追溯。

4.  质量控制：检测不合格或参数超限时，Lot 自动进入 Hold，后续由质量/工艺人员决定返工、报废或偏差放行。

5.  追溯闭环：任何一次进站、出站、检测、异常、返工、物料消耗、设备参数都必须能够按 Lot/SN 查回。

# 4. AI 应用模块：做到“能演示、能讲清楚、能防追问”

**重点：**你不是做算法训练，而是做企业 AI 应用工程化。面试官会关心：数据从哪里来、怎么检索、怎么限制幻觉、怎么输出结构化结果、怎么留痕、怎么保证安全。

| **AI 功能** | **输入数据** | **输出结果** | **工程亮点** |
|----|----|----|----|
| AI 良率分析报告 | 日期、产线、工单、Lot、良品数、不良数、缺陷 TopN、设备异常。 | 良率波动摘要、主要缺陷、疑似工序/设备、改进建议。 | 把结构化 MES 数据转成业务报告，输出 JSON 并留痕。 |

## 4.3 自然语言查 SQL 的安全边界

- 只允许 SELECT，禁止 INSERT/UPDATE/DELETE/DROP/ALTER/TRUNCATE。

- 只开放只读视图，例如 v_yield_daily、v_defect_top、v_equipment_alarm。

- 做表白名单和字段白名单，拒绝查询用户、权限、密钥等敏感表。

- 强制 LIMIT，例如最多 100 条。

- 设置 SQL 超时，避免复杂查询拖垮数据库。

- 执行前展示 SQL 审核结果；项目演示中可只做“生成 SQL + 人工确认 + 查询”。

## 5.2 MES 业务高频问题

| **问题** | **回答抓手** |
|----|----|
| MES 是解决什么问题的？ | 连接 ERP/计划系统与车间现场，控制生产执行，记录从原料到成品的过程数据。 |
| MES 和 ERP 的区别？ | ERP 管计划、订单、财务和资源；MES 管车间执行、工序流转、设备、质量、追溯。 |
| 工单和 Lot 有什么区别？ | 工单是生产任务，Lot 是现场流转对象；工单可拆成多个 Lot。 |
| 什么是 Track In / Track Out？ | Lot 进入工序/设备时进站，加工完成后出站，并记录时间、设备、Recipe、检测结果。 |
| 为什么要做 Route 控制？ | 防止跳站、漏站、错站，保证产品按正确工艺路线生产。 |
| Recipe 为什么重要？ | 它是设备加工参数包，错误 Recipe 会直接影响质量和良率。 |
| Hold 的业务意义？ | 不是普通状态字段，而是异常控制手段；Lot 被 Hold 后不能继续流转。 |
| Rework 怎么设计？ | 选择返工原因和返工路线，记录返工前后状态，进入追溯链。 |
| 追溯怎么做？ | 按 SN/Lot 查工单、工序、设备、Recipe、参数、物料、检测、异常和返工记录。 |
| 良率下降怎么排查？ | 按时间、产线、工序、设备、Recipe、不良代码逐层聚合，定位异常集中点。 |

## 5.4 AI 应用高频追问

| **问题** | **回答抓手** |
|----|----|
| 你做的是 AI 还是调接口？ | 定位为 AI 应用工程化：把业务数据、知识库、模型调用、结构化输出、留痕和权限接进 Java 系统。 |
| RAG 是什么？ | 先从外部知识库检索相关资料，再把检索结果作为上下文给模型回答，用来提升事实性和上下文相关性。 |
| 为什么不用微调？ | 项目主要是企业知识问答和业务辅助分析，知识更新频繁，用 RAG 成本更低、更新更快、可追溯。 |
| 怎么减少幻觉？ | 只基于检索片段和业务数据回答；返回引用；依据不足提示人工确认；敏感操作只建议不执行。 |
| Embedding 和向量库做什么？ | 把文本转成语义向量，用户提问时做相似度检索，找出相关 SOP/故障手册片段。 |
| AI 输出怎么接入系统？ | 要求模型输出固定 JSON，用 Java DTO 接收，校验字段，再保存 ai_report_record。 |
| 自然语言查 SQL 有什么风险？ | SQL 注入、越权、误操作、慢查询；所以只允许白名单只读视图 + SELECT + LIMIT + 审核。 |
| AI 设备异常分析怎么落地？ | 取设备报警、当前 Lot、历史维修、质量异常，再检索 SOP，最后输出排查建议和引用依据。 |

# 7. 接下来 21 天执行重点：先把面试资料和项目闭环做出来

| **阶段** | **目标** | **每天最低任务** | **验收标准** |
|----|----|----|----|
| 第 1-3 天 | 补 MES 业务模型 | 每天整理 5 个 MES 概念 + 1 张流程图。 | 能说清工单、Lot、Route、Recipe、Hold、追溯。 |
| 第 4-7 天 | 项目主数据 + Route | 每天写 2-3 个接口。 | 产品、工序、设备、Route、Recipe 可增删改查。 |
| 第 8-12 天 | 工单 + Lot + 过站 | 每天完成一个主流程接口。 | 创建工单、生成 Lot、Track In、Track Out 跑通。 |
| 第 13-15 天 | 质量 + Hold + 追溯 | 每天完成一个业务闭环。 | 检测不合格 → Hold → 处理 → 追溯可查。 |
| 第 16-18 天 | AI 良率/异常分析 | 每天完成一个 AI 接口或 Prompt。 | 能生成结构化 JSON 报告并保存。 |
| 第 19-21 天 | RAG 知识库 + 面试材料 | 每天 1 小时项目，1 小时面试话术。 | RAG 问答可演示；自我介绍和项目介绍能背。 |

**低自控力执行规则：**每天只要求完成“最低任务”，比如 1 个接口、1 张表、1 个面试题回答。状态好再加量。断一天不补罚，第二天只重启最低任务。目标是不断线，不是每天完美。
